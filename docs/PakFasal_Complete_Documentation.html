1. Project Purpose (Beginner Summary)

PakFasal is a smart-farming companion app aimed at Pakistani farmers. It combines:

  • Account access (Firebase email/password, optional guest mode).
  • Weather from external HTTP APIs with local caching.
  • Learning videos via YouTube Data API when a key is configured, else demo data.
  • Crop calendar timelines from a static Punjab-focused catalog.
  • Marketplace browsing of demo products (filter/search/favourites only).
  • Sensors + DSS: manual entry of soil moisture and pH; rules combine Firestore thresholds, weather rain chance, and client-side logic to produce irrigation/soil/risk/monitoring text; history syncs to Firestore.
  • AI chat UI as a prototype (voice input works; answers are stubbed).

  11. AuthSessionController

File: lib/src/core/routing/app_routes.dart

AuthGate (lib/src/core/widgets/auth_gate.dart):

  • Watches AuthSessionController.
  • If isSignedIn → shows protected child (e.g. HomeDashboardScreen).

6. Module Reference

6.1 Authentication

Purpose: Email/password sign-in and registration via Firebase Authentication; password reset email; optional Continue as guest; profile display name updates.

Frontend flow:

  1. Splash (splash_screen.dart): After ~5s, reads Hive app_preferences → onboarding_seen. If false → onboarding. Else if AuthSessionController.isSignedIn → home, else login.
  2. Login / Signup / Forgot password screens use forms and call controller methods.
  3. After success, navigation typically pushNamedAndRemoveUntil → /home.

Backend flow: All auth operations go through Firebase Auth SDK (FirebaseAuth.instance). No custom REST “auth API” in this repo.

Database: Firebase Auth stores users in Firebase’s identity backend (not Firestore collections for credentials). The app does not mirror user profiles to Firestore in code.

State: AuthSessionController (ChangeNotifier): _auth.authStateChanges(), loading/error mapping via _mapAuthError, guest flag.

Validation: Form validators on screens (email format, password length/strength patterns where implemented — see auth_form_provider.dart if used).

Error handling: Exceptions mapped to localization keys (invalidEmail, wrongPassword, etc.).

6.2 Weather Module

Purpose: Show current conditions, hourly slider, daily forecast, alerts (when API provides them), farmer advisory hints, city search, saved cities.

Workflow (step by step):

  1. WeatherProvider.ensureLoaded() or refreshAll() runs.
  2. WeatherRepository.resolveActiveLocation():
  3. If user previously picked a city (SharedPreferences weather.selected_location) and it is not “current GPS”, use it.
  4. Else try GPS via LocationService → reverse geocode label via API.
  5. Else use last cached snapshot location.
  6. Else default Lahore, Pakistan.
  7. WeatherRepository.fetchSnapshot() checks Hive cache key snapshot_{lat_lon}:
  8. If fresh (< 10 minutes TTL for snapshot — see repository _isFresh) and not forcing refresh → return cached.
  9. Else WeatherApiService.fetchSnapshot() calls external HTTP:
  10. If OPENWEATHER_USE_ONECALL_V3=true and API key → OpenWeather One Call 3.0.
  11. Else if OpenWeather API key → 2.5 current + 5-day/3-hour + air pollution.
  12. Else → Open-Meteo (no key) + Open-Meteo air quality helper.
  13. On success, snapshot JSON serialized to Hive; prefs updated.
  14. On failure, stale cache served if available (fromCache, isStale flags).

Frontend: WeatherScreen, CitySearchScreen, HomeWeatherCard consume WeatherProvider via Consumer/context.watch.

Caching:

  • Hive weather_cache box: keys snapshot_{lat_lon} — full WeatherSnapshot JSON.
  • SharedPreferences: selected location, saved locations list (max 5), last sync millis.

API endpoints (representative):

Provider Example endpoint pattern
Open-Meteo forecast GET {WEATHER_API_BASE_URL}/forecast?latitude=&longitude=&current=...&hourly=...&daily=...
Open-Meteo geocode GET https://geocoding-api.open-meteo.com/v1/search?name=
Open-Meteo reverse GET .../reverse?latitude=&longitude=
OpenWeather current GET /data/2.5/weather?lat=&lon=&appid=
OpenWeather forecast GET /data/2.5/forecast?...
OpenWeather One Call GET /data/3.0/onecall?...
OpenWeather geo GET /geo/1.0/direct and /reverse

Request/response: Standard HTTP GET + JSON. Parsed into domain entities (CurrentWeather, DailyForecast, etc.).

Farmer advisory: farmer_advisor.dart + WeatherConstants thresholds (rain %, wind km/h, heat/frost/UV) applied to current snapshot for banners/text.

Auto-refresh: WeatherProvider.startAutoRefresh() uses WeatherConstants.autoRefreshInterval (15 minutes).

6.3 Crop Calendar

Purpose: Display season timeline (sowing → irrigation → fertilizer → harvest, etc.) for crops Wheat, Rice, Cotton across areas Multan / Lahore (Punjab-oriented month ranges).

How “recommendations” are generated: Not AI. CropCalendarCatalog holds static CropCalendarPlan objects with MonthRange windows. CropCalendarRepository.loadPlan() picks exact (crop, area) or falls back to another area. CropCalendarProvider computes currentStageIndex and seasonProgress from today’s date vs those month ranges.

Database: None — all in Dart constants.

APIs: None.

6.4 Marketplace

Purpose: Browse agricultural products (demo), filter by category/company, search, favourites.

Data source: mock_products.dart — compile-time list of Product entities.

Provider: MarketplaceProvider — filters mockProducts, toggles favourites in memory only.

Upload / fetch from server: Not implemented.

6.5 Sensor Integration & DSS (Decision Support)

Purpose: Capture manual soil moisture (0–100%) and pH (0–14); show trends; combine rain probability from weather module; apply threshold rules (defaults + optional Firestore overrides); persist readings and show structured DSS cards (irrigation, soil, risk, monitoring).

How sensors “send data” in this app:

  • There is no Bluetooth/Wi-Fi sensor driver in code. The user types values in a form.
  • On submit, SensorRepository.addReading() writes to Firestore collection sensor_readings with FieldValue.serverTimestamp().
  • watchRecentReadings() listens to the latest 20 documents ordered by timestamp, maps to SensorReading.

Firestore collections:

Collection Document Fields used
sensor_readings auto-id soilMoisture, phLevel, crop, rainChancePercent, recommendationSummary, recommendationDetails, recommendationPriority, timestamp
sensor_config rules Nested maps: default, Wheat, Rice, Cotton with threshold doubles/ints (see SensorRepository.fetchRuleSet)

Admin / control: An administrator updates sensor_config/rules in Firebase Console (acts as remote config for thresholds). If missing, SensorRuleSet.fallback() supplies agronomy-style defaults.

DSS logic: Implemented in sensor_screen.dart (_buildRuleBasedRecommendation) — rule IDs like M1–M4 (moisture/rain), P1–P3 (pH). Priority scored in _calculatePriority → HIGH / MEDIUM / LOW.

Weather coupling: _loadCurrentRainChance() uses WeatherRepository.fetchCurrentWeather() so rain chance aligns with the same weather pipeline as the rest of the app.

6.6 Learning Articles / Videos

Purpose: Educational videos per crop category; optional crop diseases catalog for browsing.

YouTube: YouTubeLearningRepository calls:

GET {youtubeApiBaseUrl}/search?part=snippet&type=video&q=Pakistan+farming+{category}&key=...

If no API key, demo videos are returned and cached in Hive learning_cache as JSON (source: demo vs live).

Catalog: crop_diseases_catalog.dart — static disease entries for navigation/detail screens.

6.7 AI Query (“Ask AI”)

Purpose: Chat-style UI with speech-to-text input and text-to-speech readback.

Behaviour: On send, after 2 seconds a bot message is appended using localized string aiPendingResponse — no external AI API.

6.8 Profile

Purpose: View/edit display name, biometric toggle placeholders, sign-out. Uses AuthSessionController and secure storage/local auth packages from pubspec.yaml as wired in that screen.

6.9 Notifications

Purpose (UI only): Home app bar opens a bottom sheet listing three hard-coded strings (irrigation reminder, weather alert, marketplace offer). Badge shows “3” statically.

No FCM, no local notification scheduler in dependencies for messaging.

6.10 Onboarding

Purpose: First-run experience; sets onboarding_seen in Hive app_preferences.

12. System Architecture Summary

  • Single Flutter client talking to Firebase Auth, Firestore (sensors + remote DSS config), and public HTTP APIs (weather, YouTube).
  • Heavy client-side logic for DSS when manual readings submitted.
  • Offline-first weather via Hive + stale indicators.

13. Data Flow Summary

  1. Weather: GPS/prefs → Repository → (cache hit OR HTTP) → WeatherProvider → widgets.
  2. Learning: Repository → (Hive OR YouTube OR demo) → screens.
  3. Sensors: Form → rule engine + weather rain → Firestore write → stream updates charts.
  4. Auth: UI → Firebase Auth → AuthSessionController listeners → gated routes.

14. Developer Documentation — Operations

Run (see README):

flutter pub get
cp config/app_config.json config/dev.json   # then edit keys
flutter run --dart-define-from-file=config/dev.json

Secrets: Never commit real keys; use config/dev.json (gitignored) or CI dart-define.

Firestore indexes: For sensor_readings ordered by timestamp, create a composite index in Firebase Console if queries require it (the code uses orderBy('timestamp')).

15. Future Improvements (Concrete)

  1. Backend marketplace: REST API + Firestore/Cloud Storage for real listings and image uploads; replace mock_products.dart.
  2. Real AI: Integrate Gemini/OpenAI/self-hosted endpoint; replace stub in AiQueryScreen.
